# Contributing to ccusage

Thanks for your interest in improving **ccusage**. Contributions of all sizes are
welcome — bug reports, fixes, docs, and focused features. This guide explains how to set
up a development environment, how we name branches and write commits, and what we expect
from a pull request.

## Scope

ccusage is a keyboard-first, interactive terminal app that shows your **Claude Code**
usage across all sessions: per-model token counts and API-equivalent dollar cost parsed
from local transcripts, a usage heartbeat, rolling spend windows, and the official 5-hour
and 7-day subscription limits.

**In scope:** the TUI and its keyboard navigation, the transcript parser, the cost model
and pricing table, the heartbeat, rolling-window math, the reversible statusline capture,
and the self-update path (`--update` / `--check-update`).

**Out of scope:** the OAuth `/api/oauth/usage` endpoint, multi-user or remote operation,
and historical charts beyond the current windows. These are intentionally not part of the
tool.

Two hard product rules apply to every change and are non-negotiable:

- **Never read or transmit credentials**, and make **no network calls on the panel/data
  path.** The only network access is the explicit, user-invoked `--update` /
  `--check-update`.
- **Treat `~/.claude` transcripts as read-only.** The only files the tool may modify are
  the statusline settings/script, and only reversibly, with backups.

## Development setup

ccusage targets **Python 3.10+**. Work inside a virtual environment and install the
package in editable mode with its dev extras:

```bash
python3 -m venv .venv
.venv/bin/python -m pip install -e .[dev]
```

> **zsh users:** quote the extras — `.venv/bin/python -m pip install -e '.[dev]'` —
> otherwise zsh treats `.[dev]` as a glob and the install fails with `no matches found`.

Run the test suite:

```bash
.venv/bin/python -m pytest -q
```

You can run the app from source with the bundled launcher (`./ccusage`) or via the
`ccusage` entry point that the editable install places in `.venv/bin/`.

## Branch naming

Create a branch off `main` named `type/short-kebab-desc`, where `type` is one of the
allowed types below and the description is a few kebab-case words. Examples:

```
fix/heartbeat-axis
feat/csv-export
docs/contributing-guide
```

Allowed `type`s: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`, `release`.

## Commit messages

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```
type(scope): subject

Body explaining why the change is needed (what problem it solves),
not just what changed. The diff already shows what changed.
```

- **Subject:** `type(scope): subject`, written in the **imperative mood** ("add", not
  "added" or "adds"), **≤ 72 characters**. The `scope` is optional but encouraged (e.g.
  `feat(heartbeat): ...`).
- **Blank line**, then a **body** that explains **why** — the motivation, the trade-offs,
  anything a future reader needs to understand the change.
- **Allowed types:** `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`,
  `release`.

## Pull requests

- The **PR title follows the commit convention** (`type(scope): subject`).
- The **description must state what, why, and how it was tested.**
- **All CI checks must pass before merge** (see [Code style & tests](#code-style--tests)).
- **Keep PRs focused** — one logical change per PR. Smaller PRs are easier to review and
  merge.

For anything that changes the version, the release flow, or the changelog, follow
[VERSIONING.md](VERSIONING.md) rather than improvising — it is the source of truth for the
version policy and release process.

## No AI-attribution notes (hard rule, enforced by CI)

Commit messages and PR descriptions **must not** contain machine-generated
authorship/attribution lines. A CI check (`commit-policy.yml`) enforces this on the new
commits in each push or pull request, **case-insensitively**. The following are
explicitly banned:

- `Co-Authored-By:` lines naming an AI/assistant (Claude, ChatGPT/GPT/OpenAI, Copilot,
  Gemini/Bard, Cursor, …) **or** a bot account (any `…[bot]` co-author line, e.g. a
  `Co-Authored-By: dependabot[bot]` line)
- `Generated with/by <AI tool>` — an AI tool must be named (Claude, GPT, Copilot, Cursor, …)
- "Co-authored with Claude/ChatGPT/Copilot/Cursor"
- the 🤖 robot marker
- bot commit emails: the literal `noreply@anthropic.com`, **or** any
  `…[bot]@users.noreply.github.com` bot address

A couple of practical notes so the check doesn't surprise you:

- **`generated with` / `generated by` only trips when an AI tool is named** (Claude, GPT,
  Copilot, Cursor, …). Innocent phrasing like "regenerate the client (generated with
  openapi-generator)" passes.
- The vendor/bot tokens above (`claude`, `gpt`, `gemini`, `[bot]`, …) are matched as
  **case-insensitive substrings *within* a `Co-Authored-By:` line** — not as whole
  words. In the rare case that a real co-author's name or handle happens to contain one
  of these substrings (say a `Co-Authored-By:` line containing `gpt`), the check will
  flag it; reword or drop that trailer. Note the tokens only fire inside a
  `Co-Authored-By:` line, so the same substring in ordinary prose is harmless.
- **Human GitHub privacy emails are fine.** Only bot addresses
  (`…[bot]@users.noreply.github.com`) are rejected; a normal
  `12345+you@users.noreply.github.com` privacy email passes.

**Rationale:** a human author is accountable for every contribution. Tooling is welcome to
help you write code, but the commit and PR record should reflect the person responsible
for the change, not the tool that assisted it.

The two commits already on `main` predate this policy; the CI check scopes to **new**
commits only and will not retroactively fail that history.

## Code style & tests

- **Match the surrounding style.** Keep the diff legible; no dead code or speculative
  abstractions.
- **New behaviour needs a test.** Add or update tests alongside your change.
- **Tests must pass** (`.venv/bin/python -m pytest -q`) before you open a PR, and CI runs
  the suite across the supported Python versions.
